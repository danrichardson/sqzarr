## Description
Adding multiple directories with the same settings (min age, max bitrate, min size) is tedious — each one must be configured individually. The user wants a way to either:

1. **Copy/preset**: Copy settings from an existing directory config and apply them to multiple new directories at once (like a preset/template).
2. **Subdirectory support**: Add multiple subdirectories within a parent that inherit the parent's settings.

For example, `/media/Videos/Backlog` has settings (min age: 7, max bitrate: 2222000, min size: 500MB). The user wants to quickly apply those same settings to many other directories without re-entering each field every time.

This has been requested before and not implemented.

## Investigation
<!-- Agent fills this during /ticket-investigate -->

**How directories are stored:** Directories live exclusively in SQLite (`directories` table in `internal/db/db.go` lines 78-87). The TOML config (`sqzarr.toml`) only stores `root_dirs` — the allowed filesystem roots for browsing — not individual scan directories. There is no concept of templates, presets, or inheritance anywhere in the codebase.

**Directory fields:** `id`, `path` (unique), `enabled` (bool), `min_age_days` (int), `max_bitrate` (int), `min_size_mb` (int), `created_at`, `updated_at`. The DB struct is in `internal/db/directories.go` (`type Directory`). The API input struct is in `internal/api/handlers.go` `handleCreateDirectory` (snake_case JSON). The frontend mirrors this with `Directory` and `DirectoryInput` interfaces in `frontend/src/lib/api.ts`.

**API endpoints:** Standard CRUD — `GET /directories`, `POST /directories`, `GET /directories/{id}`, `PUT /directories/{id}`, `DELETE /directories/{id}`. Create accepts a single directory at a time; path validation checks against `root_dirs`, verifies the directory exists and is writable.

**Frontend UI:** `frontend/src/pages/Directories.tsx` already supports bulk add — the "Add Directories" form maintains an `addPaths: string[]` array with shared `addSettings: Settings` applied to all paths. It loops over paths calling `api.createDirectory` one at a time. However, there is no way to pre-populate `addSettings` from an existing directory. The `defaultSettings` are hardcoded (`min_age_days: 7`, `max_bitrate: 2_222_000`, `min_size_mb: 500`).

**What's missing for TKT-005:**
1. No "Copy settings" action on existing directory cards — no way to seed the add form with an existing directory's settings.
2. No backend batch-create endpoint (the frontend already loops, but a batch endpoint would be cleaner and atomic).
3. No preset/template persistence (not strictly needed — "copy from existing" is simpler and avoids new DB tables).

Regression Risk: low


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

**Approach: "Clone settings" from existing directory + use existing bulk-add form.** This avoids new DB tables, new backend concepts, and complex template CRUD. The bulk-add UI already exists — we just need to let users seed it from an existing directory.

**Frontend changes (Directories.tsx):**
- Add a "Copy settings" button (clipboard/copy icon) to each directory card in the list, next to the existing Edit and Delete buttons.
- When clicked, it opens the existing "Add Directories" form with `addSettings` pre-populated from that directory's `min_age_days`, `max_bitrate`, `min_size_mb`, and `enabled` values (instead of `defaultSettings`).
- Add a small label at the top of the add form: "Settings copied from: /path/to/source" so it's clear which directory the settings came from.

**Backend changes (optional but recommended):**
- Add a `POST /directories/batch` endpoint that accepts `{ paths: string[], settings: { enabled, min_age_days, max_bitrate, min_size_mb } }` and inserts all directories in a single transaction. This replaces the frontend's current loop-over-paths approach, making bulk add atomic (all-or-nothing) and reducing HTTP round trips.
- Add a corresponding `InsertDirectories(dirs []*Directory) ([]int64, error)` method in `internal/db/directories.go` that uses a transaction.

**Why not full presets/templates:** Adding a `presets` table, preset CRUD UI, and preset-selection dropdown is significantly more work for marginal benefit. Most users have 2-3 distinct setting profiles; copying from an existing directory covers that need without new persistence or management UI. If demand grows, presets can be layered on later.

**Why not subdirectory inheritance:** Inheritance would require a parent-child relationship model, recursive resolution logic, and UI for showing inherited vs. overridden values. It's a much larger feature that can be considered separately. The "copy settings" approach is simpler and solves the stated problem.
