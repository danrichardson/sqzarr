## Description
Currently, changing the password requires running `sqzarr hash-password` on the command line and manually editing `sqzarr.toml`. The settings panel even says this explicitly. This is a poor UX — the user should be able to change their password directly from the settings UI like any modern web application.

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### Current authentication flow

1. **Password storage**: The password hash is stored in `sqzarr.toml` under `[auth]` as `password_hash` (bcrypt). The `AuthConfig` struct in `internal/config/config.go` holds `PasswordHash` and `JWTSecret`.

2. **Login endpoint**: `POST /auth/login` in `internal/api/handlers.go` (line 527). Accepts `{"password": "..."}`, verifies against `s.cfg.Auth.PasswordHash` using `bcrypt.CompareHashAndPassword`, and returns a JWT valid for 30 days signed with `s.cfg.Auth.JWTSecret` (HS256).

3. **Auth middleware**: `internal/api/middleware.go` — if `PasswordHash` is empty, auth is disabled entirely. Otherwise, all routes except `POST /auth/login` require a valid JWT Bearer token (or `?token=` query param for SSE/WS).

4. **`sqzarr hash-password` CLI**: `cmd/sqzarr/main.go` `runHashPassword()` (line 241). Reads plaintext from stdin, runs `bcrypt.GenerateFromPassword` with `DefaultCost`, prints `password_hash = "<hash>"` for manual copy-paste into `sqzarr.toml`.

5. **Hashing library**: `golang.org/x/crypto v0.31.0` — specifically `golang.org/x/crypto/bcrypt`. Already imported in both `cmd/sqzarr/main.go` and `internal/api/handlers.go`.

### Config update mechanism

The project already supports runtime config updates with TOML file write-back:

- `internal/config/update.go` provides `UpdateFile(path, map[string]string)` which does regex-based in-place key replacement in the TOML file, preserving comments and structure. Unknown keys are inserted into the correct `[section]` by lookup in `keySection`.
- `PUT /config` in `handleUpdateConfig` (line 584) updates the in-memory `s.cfg` fields and then calls `config.UpdateFile(s.cfgPath, fileUpdates)` to persist to disk — same pattern used by `persistPaused`.
- The `keySection` map in `update.go` does **not** currently include `password_hash` or `jwt_secret`, but adding them is trivial (map them to `"auth"`).

### What needs to change

- A new `POST /auth/change-password` endpoint that: verifies the current password, hashes the new one with bcrypt, updates `s.cfg.Auth.PasswordHash` in memory, and writes back to `sqzarr.toml` via `config.UpdateFile`.
- The endpoint must be behind auth middleware (user must be logged in).
- The `keySection` map needs `"password_hash": "auth"` added.
- The frontend Settings page needs a password change form replacing the current "run sqzarr hash-password" text.

### Risks and edge cases

- **Concurrent config writes**: `config.UpdateFile` does a non-atomic read-modify-write. This is the existing pattern for all config updates — no new risk introduced.
- **JWT invalidation**: Changing the password does not invalidate existing JWTs (they are signed with `JWTSecret`, not the password hash). This is acceptable — tokens expire after 30 days. Optionally, rotating `JWTSecret` on password change would force re-login, but that is out of scope.
- **No DB migration needed**: Password stays in TOML, consistent with current architecture.

Regression Risk: low


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

**Keep password in TOML with file write-back** (not DB). This is the right call because:

1. The config update infrastructure already exists (`config.UpdateFile`) and is battle-tested by `PUT /config` and `persistPaused`.
2. The password hash is a single-user credential that belongs with the rest of the server config, not in the jobs database.
3. Moving to DB would require a migration, a new table, and changing the auth middleware's config source — all unnecessary complexity for a single-user app.

### Backend

Add `POST /auth/change-password` accepting `{"current_password", "new_password"}`:
1. Verify `current_password` against `s.cfg.Auth.PasswordHash` with `bcrypt.CompareHashAndPassword`.
2. Hash `new_password` with `bcrypt.GenerateFromPassword` (DefaultCost).
3. Update `s.cfg.Auth.PasswordHash` in memory.
4. Write `password_hash` to `sqzarr.toml` via `config.UpdateFile`.
5. Return 200 on success.

### Frontend

Replace the read-only "run sqzarr hash-password" card in `Settings.tsx` with a form containing: current password, new password, confirm new password. Validate confirm match client-side, call the new endpoint, show success/error feedback.
