## Description
The main dashboard has a "space saved" block that displays the total disk space saved by transcoding. Currently it's static text. It should be clickable and lead to a detail view that shows how that number was calculated ‚Äî i.e., a breakdown of individual files showing original size, transcoded size, and savings per file.

## Investigation
<!-- Agent fills this during /ticket-investigate -->

**Dashboard rendering:** The "Space Saved" stat is rendered in `frontend/src/pages/Dashboard.tsx` (line 178-183) inside a `<Card>` component. It displays `formatBytes(status.total_saved_gb * 1024 * 1024 * 1024)` as static text. There is no click handler or link.

**Data source (API):** The `GET /status` handler in `internal/api/handlers.go` (line 21-53) calls `s.db.GetStats()` which returns `TotalBytesSaved`. This is computed live from the `jobs` table in `internal/db/stats.go` (line 15-28) via:
```sql
SELECT COALESCE(SUM(CASE WHEN status IN ('done','staged') THEN COALESCE(bytes_saved,0) ELSE 0 END), 0) ...
FROM jobs
```

**Per-file data available:** The `jobs` table already stores everything needed for the breakdown:
- `source_path` ‚Äî file name
- `source_size` ‚Äî original file size in bytes
- `output_size` ‚Äî transcoded file size in bytes (nullable)
- `bytes_saved` ‚Äî pre-computed savings per job (nullable)
- `status` ‚Äî only `done` and `staged` jobs contribute to the total

The existing `GET /jobs?status=done` and `GET /jobs?status=staged` endpoints (via `ListJobs` in `internal/db/jobs.go`) already return full `Job` structs including `SourcePath`, `SourceSize`, `OutputSize`, and `BytesSaved`. No new database query is strictly required, but a dedicated endpoint would be cleaner and more efficient (fetching only the fields needed, combining both statuses in one call, and including a running total for verification).

**Frontend routing:** The app uses `react-router-dom` with `createBrowserRouter` in `frontend/src/App.tsx`. Routes are nested under a `<Layout>` element that provides sidebar navigation. The SPA handler in `server.go` already serves `index.html` for unknown paths, so adding a new client-side route like `/savings` will work without backend changes. The History page (`frontend/src/pages/History.tsx`) already demonstrates the pattern for rendering per-job savings with expandable rows, `formatBytes`, and `timeAgo` utilities.

**Regression Risk: low** ‚Äî This is a new page and a minor UI change to an existing card. No existing functionality is modified beyond wrapping a `<Card>` in a `<Link>`.


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

**Approach: new `/savings` route with a dedicated API endpoint.**

1. **Backend ‚Äî new `GET /jobs/savings` endpoint** that returns all jobs with `status IN ('done','staged')` and `bytes_saved > 0`, selecting only the columns needed for the breakdown: `id`, `source_path`, `source_size`, `output_size`, `bytes_saved`, `finished_at`. Ordered by `bytes_saved DESC` so the biggest wins appear first. This avoids reusing the generic `/jobs` endpoint (which requires two calls for two statuses and returns many unneeded fields).

2. **Frontend ‚Äî new `SpaceSaved` page** at `/savings` that fetches the new endpoint and renders a table/list showing each file's original size, new size, and savings (both bytes and percentage). Include a summary row at the top showing the grand total (which should match the dashboard stat). Reuse existing `formatBytes` utility and the `Card` component for consistent styling.

3. **Dashboard ‚Äî make the Space Saved card clickable** by wrapping it with a `<Link to="/savings">` from `react-router-dom`. Add a subtle hover effect (e.g., `cursor-pointer hover:border-amber-300`) to signal interactivity.

4. **Router ‚Äî add the `/savings` route** in `App.tsx` under the `<Layout>` parent. No sidebar nav entry needed ‚Äî this is a detail view reached only from the dashboard card.
