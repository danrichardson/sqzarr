## Description
After files are reviewed in the Review page, the review count badge in the sidebar does not reset to zero and the "Space Saved" metric on the Dashboard does not update to reflect the reviewed files. These may be two separate issues or share a common root cause.

## Reproduction Steps (bugs only)
1. Have files pending in the Review queue (e.g., 29 shown in sidebar badge)
2. Review/approve the pending files
3. Observe the sidebar badge still shows the previous count instead of resetting
4. Navigate to the Dashboard and observe "Space Saved" has not changed

## Acceptance Criteria
- [ ] After all review items are approved/dismissed, the sidebar review count badge resets to 0
- [ ] After reviewed files are accepted, the "Space Saved" dashboard metric updates to reflect the new totals
- [ ] Both the review count and space saved update without requiring a page refresh

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### Bug 1: Review count badge does not update

**Root cause**: The sidebar badge count is fetched once on mount and never refreshed.

In `frontend/src/components/Layout.tsx` (lines 21-28), the `originalsCount` state is set by a single `useEffect` with no dependencies:

```ts
useEffect(() => {
  api.listOriginals()
    .then(r => setOriginalsCount(r?.length ?? 0))
    .catch(() => {})
  // ...
}, [])
```

This runs once when `Layout` mounts. When the user navigates to the Review page and deletes/restores originals, the Review page updates its own local state via its `load()` callback (`frontend/src/pages/Review.tsx` lines 28-37), but the Layout component's `originalsCount` is completely independent. There is no shared state, no context, no event bus, and no polling to propagate changes from the Review page back to the sidebar.

The SSE system (`internal/queue/worker.go` lines 23-30) only emits `progress`, `done`, `failed`, and `paused` events — there is no event type for review actions (original deleted/restored).

### Bug 2: Space Saved does not update after review

**Root cause**: This is partially a non-issue and partially a stale-data problem.

The `GetStats()` query in `internal/db/stats.go` (lines 18-23) computes `TotalBytesSaved` by summing `bytes_saved` from jobs with status `done` OR `staged`:

```sql
COALESCE(SUM(CASE WHEN status IN ('done','staged') THEN COALESCE(bytes_saved,0) ELSE 0 END), 0)
```

When a transcode completes, the job status is set to `staged` and `bytes_saved` is populated (`internal/db/jobs.go` lines 195-204 via `StageJob`). This means space saved is already counted *before* the user reviews and accepts the file. Accepting (deleting the original) changes the job from `staged` to `done` — both are counted, so the total doesn't change. This matches correct behavior: the space is saved as soon as the smaller file replaces the larger one.

However, when a user **restores** an original (reverts), the job status becomes `restored` (line 1093 of `handlers.go`), which is correctly excluded from the stats sum. The Dashboard polls status every 3 seconds (`frontend/src/pages/Dashboard.tsx` lines 78-84), so a restore should be reflected within a few seconds — but only if the user is on the Dashboard page. If they're on the Review page, the Dashboard component is unmounted and no polling occurs. When they navigate to Dashboard, the `load()` on mount fetches fresh data, so it should show the correct value.

The real space-saved issue is: the Dashboard's 3s polling only runs while the Dashboard page is mounted. If the user reviews files on the Review page and navigates to Dashboard, `load()` on mount will fetch correct data. So the value *should* be correct on navigation. If the user reported this as a bug, the likely scenario is that the space saved number was already counting staged jobs, and the user expected it to increase upon acceptance — a UX expectation mismatch, not a data bug.

**However**, to satisfy the acceptance criteria ("Space Saved updates after reviewed files accepted"), we should ensure the Dashboard refreshes stats when navigated to after review actions. The current `load()` on mount already does this. The fix for completeness: add the originals count refresh to the same mechanism used for bug 1.

### Are they connected?

Yes. Both stem from the same architectural gap: the Layout component has no mechanism to learn about state changes that happen in child pages. The fix is a shared state invalidation mechanism.

## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

**Approach**: Use a lightweight React context with a callback-based invalidation pattern. When a review action occurs on the Review page, it calls an `invalidate()` function provided by the Layout context. The Layout re-fetches the originals count (and optionally status) in response.

This is preferred over:
- **Polling in Layout** — wastes API calls for a rarely-changing count; adds unnecessary load.
- **Global state library (Redux/Zustand)** — overkill for this single piece of shared state; this project doesn't use one.
- **SSE events for review actions** — would require backend changes to emit events from API handlers (they don't have access to the worker's event bus); more invasive than necessary.

**Tradeoffs**: The context adds a small amount of plumbing but keeps the solution self-contained in the frontend. The invalidation is explicit (triggered by the action) rather than implicit (polling), so it updates instantly.

For the Space Saved UX concern: since the Dashboard already fetches fresh data on mount, no backend changes are needed. The acceptance criteria says "both update without requiring a page refresh" — the sidebar badge fix addresses the review count, and the Dashboard's existing mount-time fetch addresses Space Saved. If we want Space Saved to update while staying on a *different* page (e.g., a dashboard widget in the sidebar), that would be a separate enhancement.
