## Description
Several stat blocks on the main dashboard are currently static and should be interactive:

1. **Jobs Done block** — should be clickable and show a breakdown of how that number was calculated (list of completed jobs with details).
2. **Failed block** — should be clickable and navigate to the failed items so the user can see what failed and why.
3. **Status block** — currently shows a start/stop control, but there's also a separate pause button. The purpose and distinction between these is unclear. Investigate whether both are needed or if one should be removed/consolidated.

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### How the stat blocks are rendered

The Dashboard page (`frontend/src/pages/Dashboard.tsx`) renders four stat blocks in a CSS grid at lines 177-201:

1. **Space Saved** — displays `status.total_saved_gb` formatted as bytes. Static, informational. Not in scope.
2. **Jobs Done** — displays `status.jobs_done` (an integer from `GET /status`). Currently a plain `<Card>` with text — not clickable, no link, no drill-down.
3. **Failed** — displays `status.jobs_failed` in red. Same story: plain `<Card>`, not interactive.
4. **Status** — shows a colored dot (green = Running, grey = Paused) and the word "Paused" or "Running". It is **not** a clickable control — it is purely a label. There is no start/stop toggle inside this card.

### Where the pause control actually lives

There are **two** pause/resume controls, but they are not redundant — they serve different UX roles:

- **Dashboard header button** (`Dashboard.tsx` lines 148-155): A `Pause` / `Resume` toggle button in the top-right action bar. Calls `api.pauseQueue()` or `api.resumeQueue()` and reloads status. This is the primary control for the user who is already on the dashboard.
- **Layout banner** (`Layout.tsx` lines 86-101): A global amber banner that appears across every page when the queue is paused, with a "Resume" button. Calls `api.resumeQueue()`. This is a passive notification + quick-resume affordance — it does NOT offer a "Pause" action, only "Resume".

Both call the same backend endpoints: `POST /queue/pause` and `POST /queue/resume` (in `handlers.go` lines 513-524), which delegate to `worker.SetPaused(bool)` and persist the state to config.

The **Status card** on the dashboard is purely a label — it shows paused/running state but has no click handler. It is NOT a start/stop control, contrary to the ticket description. The ticket's concern about "start/stop control" in the Status block appears to be a misunderstanding; the actual toggle is in the header action bar, not inside the card.

**Verdict: No redundancy.** The dashboard Pause button and the layout Resume banner serve complementary purposes. The Status card is display-only. No consolidation is needed.

### API data available for drill-downs

- `GET /status` returns aggregate `jobs_done` and `jobs_failed` counts (computed live from the `jobs` table in `db.GetStats()`).
- `GET /jobs?status=done&limit=N&offset=N` and `GET /jobs?status=staged&limit=N&offset=N` return completed jobs with full detail (source path, output size, bytes saved, encoder used, timestamps). The "Jobs Done" count includes both `done` and `staged` statuses.
- `GET /jobs?status=failed&limit=N&offset=N` returns failed jobs with `ErrorMessage` containing the failure reason.
- The existing **History page** (`/history`) already displays all jobs (all statuses) with expandable detail rows showing error messages, sizes, and savings. It supports pagination.

### Existing routing

`App.tsx` defines these routes: `/` (Dashboard), `/queue`, `/history`, `/directories`, `/review`, `/settings`, `/login`. Navigation is via sidebar (`Layout.tsx`). There is an SPA fallback handler on the backend so client-side routes work on refresh.

The History page already serves as a general job browser. Rather than building entirely new pages, the stat blocks can navigate to `/history` with a status filter query param, or use click-to-expand modals/drawers on the dashboard itself.

### Regression Risk: low

Changes are purely additive UI (click handlers, navigation, optional modal). No backend changes needed. The existing `GET /jobs?status=...` API already supports the required filtering.


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

### 1. Jobs Done — clickable with drill-down

Make the "Jobs Done" card clickable. On click, navigate to `/history?status=done` (or open an inline drawer/modal showing recent completed jobs). The History page already has the UI for expanding job details. Add query-param support to History so it can pre-filter by status.

### 2. Failed — clickable with drill-down

Make the "Failed" card clickable. On click, navigate to `/history?status=failed`. The History page already shows error messages in expandable rows, so no new detail UI is needed — just filtering.

### 3. Status card — clarify, do not change controls

The Status card is already display-only and is not redundant with the Pause button. Add a subtle visual cue (e.g., make it clickable to toggle pause, or add a small toggle icon) so users understand how to control the queue from the dashboard. Alternatively, leave it as-is since the Pause/Resume button is immediately adjacent in the header. **Recommendation:** leave the existing controls as-is and add a brief tooltip or "click Pause above" hint to the Status card. No consolidation needed.

### Approach: URL-based filtering on History page

This is the lightest-weight approach: no new pages, no modals, no new API endpoints. The History page reads `?status=` from the URL and pre-selects a filter. The stat cards become `<Link>` elements pointing to the filtered History view.
