## Description
FFmpeg exits with status 255 when attempting to transcode HEVC video. The system tries to use CUDA hardware acceleration but fails ungracefully when the CUDA library (libcuda.so.1) is unavailable.

**Context:** Sqzarr is running in a containerized environment that was migrated from an i7-8650u to an i5-8350u host. The container may not have the correct device pass-through for Intel VAAPI hardware encoding/decoding, or hardware detection is selecting an unavailable encoder without proper validation.

**Affected file:** One Day (2024) Season 1 Episode 1 1080p WEB h265-REVILS
- Container: MKV
- Video: HEVC
- Subtitles: 8 streams (English, Spanish, French, German, Italian, Dutch, Swedish, Ukrainian)

**Error output:**
```
[AVHWDeviceContext @ 0x5e316dc2e040] Cannot load libcuda.so.1
[AVHWDeviceContext @ 0x5e316dc2e040] Could not dynamically load CUDA
Device creation failed: -1.
[vist#0:0/hevc @ 0x5e316dc1e0c0] [dec:hevc @ 0x5e316dc1af00] No device available for decoder: device type cuda needed for codec hevc.
[vist#0:0/hevc @ 0x5e316dc1e0c0] [dec:hevc @ 0x5e316dc1af00] Hardware device setup failed for decoder: Operation not permitted
Error opening output file /media/Videos/TV/One Day (2024)/Season 1/One Day S01E01 1080p WEB h265-REVILS.sqzarr-tmp.mkv.
Error opening output files: Operation not permitted
```

**Real issue:** Host has Intel UHD Graphics (i5-8350u) which supports HEVC VAAPI encoding, but either:
1. Container device pass-through is misconfigured (missing `/dev/dri` devices)
2. Hardware detection doesn't validate actual device accessibility
3. Probe succeeds but transcode fails with cryptic errors

## Reproduction Steps (bugs only)
1. Run sqzarr in a container without proper `/dev/dri` device pass-through
2. OR move container between hosts with different GPU capabilities
3. Add an MKV file with HEVC video
4. Run transcoding job
5. Observe FFmpeg fails with cryptic device initialization errors or "Operation not permitted"

## Investigation

The root causes span hardware detection, container configuration, and user diagnostics:

1. **Weak hardware probes:** `probeNVENC()` (line 127 in [internal/transcoder/encoder.go](internal/transcoder/encoder.go)) only checks if the encoder name exists in ffmpeg output; it doesn't test actual device initialization. `probeVAAPI()` (line 69) does test the device, but provides no logging, so failures are silent.

2. **Container device pass-through not validated:** In unprivileged LXC containers (like pve2 container 260), `/dev/dri` devices must be explicitly passed through. Sqzarr has no mechanism to detect this or provide diagnostic guidance.

3. **No runtime fallback:** If hardware encoding fails mid-transcode (device unavailable, permissions changed, etc.), the job fails hard instead of retrying with software codec.

4. **Missing diagnostics:** Users deploying in containers can't see:
   - Which encoders were tested at startup
   - Why a probe failed (device missing? permissions? driver issue?)
   - Which encoder was selected and why
   - Whether device pass-through is needed

**Specific issue for pve2 container 260:**
- Host: i5-8350u with Intel UHD Graphics 620 (HEVC VAAPI capable)
- Container: Unprivileged LXC with no `/dev/dri` device pass-through visible
- Result: Hardware probe might silently fail, but user sees no explanation; software fallback should work but wasn't implemented

**Regression risk**: Low. Probe enhancements and fallback are additive. All existing behavior preserved for systems with working hardware.

## Proposed Solution

Four complementary improvements:

1. **Enhance all hardware probes** — Make NVENC probe match VAAPI's device-test rigor. Add detailed logging to all probes showing device paths tested, command executed, and pass/fail reasons.

2. **Add startup diagnostic report** — Log summary at startup listing all encoders probed, which passed/failed, which was selected. Include device paths like `/dev/dri/renderD128`.

3. **Implement runtime fallback** — Wrap ffmpeg invocation in `transcoder.Run()` with: if ffmpeg fails with device-related errors (e.g., "Could not dynamically load CUDA", "Cannot connect to display", "/dev/dri" permission denied), retry with software encoder and emit a warning log.

4. **Add documentation for containers** — Comment or doc explaining device pass-through requirements for Proxmox LXC (and Docker, if applicable). Include container config examples.

**Immediate benefit:** Sqzarr works out-of-box in unprivileged containers (software fallback). If users want hardware acceleration, logs tell them exactly what's needed.

Strengthen hardware detection, add diagnostics, implement runtime fallback, and document container deployments.

**Steps**

2. Update `probeVAAPI()`, `probeNVENC()`, `probeVideoToolbox()` to use the test helper and log pass/fail with reasons.
3. Add detailed logging to `Detect()` showing all probes attempted, results, and which encoder was selected.
4. In `transcoder.Run()`, wrap the `cmd.Wait()` error with a check: if ffmpeg stderr contains device-unavailable patterns (regex: "Cannot load|Could not dynamically load|/dev/dri|permission denied", case-insensitive), retry with software encoder.
5. Add a new doc or README section explaining Proxmox LXC device pass-through for VAAPI/NVENC (device rules, mount options, unprivileged mapping).
6. Write test cases to verify probes handle device unavailability and log correctly.
