## Description
Files that already meet (or nearly meet) the target bitrate criteria are being sent through the full transcode pipeline, only to be marked "excluded: uncompressible" after wasting time and resources. For example, a 1.5 GB h264 file gets transcoded, the output is roughly the same size, and it's flagged as uncompressible — then this repeats.

The scanner should perform a pre-flight check before queuing a file: calculate the file's current bitrate from its size and duration, compare that against the target compression criteria, and skip files that are already within a reasonable margin (e.g., target bitrate + 10%). 

The goal is to avoid wasting transcode cycles on files that are already close to optimal. We're looking to catch the files that are 5x over the target — files that are only 10% over should be left alone.

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### Current scanner flow (`internal/scanner/scanner.go`)

`ScanDirectory` walks a directory tree, skipping `.processed` dirs and non-video extensions. For each video file it calls `maybeEnqueue`, which applies these checks in order:

1. **DB dedup** — `SourcePathStatus(path)` fetches the most recent job for this source path. Files with status `pending`, `running`, `staged`, `done`, `excluded`, `skipped`, or `cancelled` are skipped. Only `failed` and `restored` statuses allow re-queuing.
2. **Output path check** — `OutputPathExists(path)` prevents re-scanning a file that is itself the output of a prior transcode.
3. **Age check** — file mtime must be older than `dir.MinAgeDays`.
4. **Size check** — file must be at least `dir.MinSizeMB`.
5. **ffprobe** — `probeFile(path)` runs `ffprobe -show_streams -show_format` and extracts the video codec name, container-level `bit_rate`, and `duration`.
6. **Bitrate check** — if `dir.MaxBitrate > 0` and `probe.bitrate <= dir.MaxBitrate`, the file is skipped with reason `"bitrate %d within limit"`.

If all checks pass, a `db.Job` is inserted with status `pending`, recording `SourceSize`, `SourceCodec`, `SourceDuration`, and `SourceBitrate`.

### Bitrate threshold configuration

- `max_bitrate` is a per-directory column in SQLite (default `2222000` bps, roughly 1 GB/hour).
- It is set/updated through the API (`POST /api/directories`, `PUT /api/directories/:id`) and surfaced in the React frontend (`Directories.tsx`).
- The scanner comparison is `probe.bitrate <= dir.MaxBitrate` — a hard threshold with no margin.

### The "uncompressible" path (`internal/verifier/verifier.go`, `internal/queue/worker.go`)

After transcoding completes, `verifier.Verify` checks whether `outputSize >= inputSize`. If so, it sets `Uncompressible = true` and `OK = false`. Back in `worker.processJob`, the uncompressible flag causes `db.ExcludeJob(job.ID, "uncompressible: "+result.Reason)`, which sets `status='excluded'` in the DB. This permanently prevents re-queuing (the scanner's `SourcePathStatus` check sees `excluded` and skips).

So files that are already near-optimal DO get excluded eventually, but only **after** a full transcode cycle has already been wasted on them.

### The gap this ticket addresses

There is no margin/tolerance on the scanner's bitrate comparison. A file at 2,300,000 bps passes the `> 2,222,000` gate, gets transcoded at `-b:v 2300k` (encoder target is 2,300,000 bps — essentially the same bitrate), the output is the same size or larger, and the verifier marks it uncompressible. The file is then excluded, but the transcode time (potentially hours for large files) has been wasted.

The encoder targets (`encoder.go`) use `-b:v 2300k` for hardware encoders and `-crf 23` for software. The hardware target of 2,300,000 bps is only ~3.5% above the default `max_bitrate` of 2,222,000, making it almost certain that files slightly above the threshold will produce same-size or larger output.

### What ffprobe already provides

The scanner already runs ffprobe and captures both `bitrate` (from `format.bit_rate`) and `duration`. No additional probing is needed to compute bitrate. The `format.bit_rate` field is the container-level average bitrate including all streams (video + audio + subs), which is appropriate for a coarse pre-flight check.

### Regression Risk: low

The change adds a configurable skip margin to an existing check. Files that previously would have been transcoded and then excluded as uncompressible will now be skipped earlier. The worst case for a misconfigured margin is skipping a file that could have been compressed — easily noticed in the UI and correctable by adjusting the margin. No changes to the transcode, verify, or file-move paths.


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

Add a configurable **bitrate skip margin** so the scanner skips files whose bitrate is within a percentage above `max_bitrate`, rather than only skipping files at or below the threshold.

**Margin logic:** A file is skipped if `probe.bitrate <= max_bitrate * (1 + margin)`, where `margin` defaults to `0.10` (10%). This means a directory with `max_bitrate = 2,222,000` would skip any file at or below ~2,444,200 bps.

**Configuration:** Add a new `bitrate_skip_margin` field to the `Directory` model (and corresponding DB column). This keeps the margin per-directory like the other thresholds, allowing different libraries to have different tolerances. Default value: `0.10`. A value of `0.0` preserves current behavior (hard threshold, no margin).

**Skip reason:** Files skipped by the margin check get logged with a clear reason like `"bitrate 2400000 within margin of limit 2222000 (threshold 2444200)"` so the UI and logs make it obvious why a file was not queued.

**Excluded files:** No changes needed — files already marked `excluded` with status `"uncompressible"` are already prevented from re-processing by the `SourcePathStatus` check in the scanner.

**No TOML config change needed.** The margin is per-directory (stored in SQLite), consistent with how `max_bitrate`, `min_age_days`, and `min_size_mb` already work. The API and frontend will expose it alongside the existing directory settings.
