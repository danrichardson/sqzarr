## Description
Files that have already been processed (or marked as excluded/uncompressible) keep reappearing in the queue on subsequent scans. It's unclear whether this is caused by clearing history in the UI resetting their tracking state, or if the scanner isn't properly checking against the database of already-handled files.

Once a file has been compressed, excluded, or otherwise handled, it should never be re-queued unless the user explicitly requests it. The system needs durable tracking that survives history clears.

Possibly related to TKT-003 (uncompressible files being retried), but this is the broader issue — any file that's been through the pipeline should be remembered permanently.

## Reproduction Steps (bugs only)
1. Let sqzarr scan and process/exclude some files
2. Clear history in the UI (if applicable)
3. Wait for the next scan cycle
4. Previously processed/excluded files reappear in the queue


## Acceptance Criteria
- [ ] Investigate how the scanner determines whether a file has already been handled
- [ ] Processed and excluded files are tracked durably (not lost when history is cleared)
- [ ] Clearing history in the UI does not cause files to be re-queued
- [ ] Files only re-enter the pipeline if explicitly requested by the user or if the source file has changed (e.g., new mtime/size)

## Investigation
<!-- Agent fills this during /ticket-investigate -->

**Root cause identified.** The scanner checks whether a file should be enqueued by calling `db.SourcePathStatus(path)` (jobs.go:296-308), which queries `SELECT status FROM jobs WHERE source_path=? ORDER BY created_at DESC LIMIT 1`. If no row exists, or the status is `failed` or `restored`, the file is eligible for re-queuing. Any other status (`pending`, `running`, `done`, `staged`, `cancelled`, `skipped`, `excluded`) causes the file to be skipped.

The `ClearHistory` method (jobs.go:233-247) deletes all jobs with status `IN ('failed','cancelled','skipped','excluded','restored')`. This means:

1. **`excluded` jobs are deleted** -- files previously marked as permanently uncompressible (e.g., via repeated failure threshold or verifier's `Uncompressible` flag) lose their tracking entirely. On the next scan, `SourcePathStatus` returns `""` (no row), so the scanner re-queues them.
2. **`cancelled` and `skipped` jobs are deleted** -- same effect; these files re-enter the pipeline.
3. **`failed` jobs are deleted** -- these were already eligible for re-queue by design, so deletion doesn't change behavior, but it does lose the `fail_count` history, resetting the failure threshold counter.

Successfully processed files (`done`, `staged`) are NOT deleted by `ClearHistory`, so they survive. However, there is no separate durable "processed files" or "excluded files" table -- the `jobs` table is the only tracking mechanism, and `ClearHistory` destroys it for excluded files.

The `OutputPathExists` check (scanner.go:136-140) provides a secondary guard for transcoded files only (it checks `jobs.output_path`), but this also relies on the `jobs` table and would fail if `done`/`staged` rows were ever cleared.

**Summary:** `ClearHistory` deletes `excluded` job records, which are the scanner's only mechanism for remembering that a file should never be re-queued. This is the direct cause of the bug described in the ticket.

Regression Risk: low

## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

Introduce a separate `processed_files` table that durably tracks every file the system has handled, independent of the `jobs` table. This table should persist across history clears.

**Key design points:**

1. **New `processed_files` table** with columns: `source_path` (unique), `status` (done/excluded/skipped), `reason` (why excluded), `source_size`, `source_mtime`, `created_at`, `updated_at`. The `source_size` and `source_mtime` fields enable re-queuing when the underlying file has genuinely changed (per acceptance criteria).

2. **Scanner uses `processed_files` as primary gate.** Before checking the `jobs` table, the scanner queries `processed_files` for the source path. If a record exists and the file's current size+mtime match, skip the file. If the file has changed (new mtime or size), delete the `processed_files` record and allow re-queuing.

3. **Write to `processed_files` on terminal job states.** When a job reaches `done`, `staged`, or `excluded` status, upsert a row into `processed_files`. This happens in the worker's `processJob` (for staged/done) and `handleFailure`/`ExcludeJob` (for excluded).

4. **`ClearHistory` leaves `processed_files` untouched.** It continues to delete terminal job rows from `jobs` for UI cleanliness, but the scanner's gate is now the durable table.

5. **Optional: explicit "reprocess" action.** Add an API endpoint that deletes a specific `processed_files` row, allowing a user to explicitly request re-processing of a single file.

- [ ] Add unit tests in `internal/db/processed_test.go` for the new DB methods
- [ ] Add test in `internal/scanner/scanner_test.go` verifying that clearing history does not cause re-queue of excluded files
