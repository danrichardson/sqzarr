## Description
Files that have already been processed (or marked as excluded/uncompressible) keep reappearing in the queue on subsequent scans. It's unclear whether this is caused by clearing history in the UI resetting their tracking state, or if the scanner isn't properly checking against the database of already-handled files.

Once a file has been compressed, excluded, or otherwise handled, it should never be re-queued unless the user explicitly requests it. The system needs durable tracking that survives history clears.

Possibly related to TKT-003 (uncompressible files being retried), but this is the broader issue — any file that's been through the pipeline should be remembered permanently.

## Reproduction Steps (bugs only)
1. Let sqzarr scan and process/exclude some files
2. Clear history in the UI (if applicable)
3. Wait for the next scan cycle
4. Previously processed/excluded files reappear in the queue


## Acceptance Criteria
- [ ] Investigate how the scanner determines whether a file has already been handled
- [ ] Processed and excluded files are tracked durably (not lost when history is cleared)
- [ ] Clearing history in the UI does not cause files to be re-queued
- [ ] Files only re-enter the pipeline if explicitly requested by the user or if the source file has changed (e.g., new mtime/size)
