## Description
When a transcode fails due to a permission denied error (e.g., ffmpeg can't write the temp output file), the job is displayed in the history tab as "Excluded" rather than "Error". This is misleading — "Excluded" implies the file was intentionally skipped (like uncompressible files), but this was an actual failure that needs user attention.

Errors like permission denied should be categorized as errors, displayed with an error status, and the user should be notified that action is required (e.g., fix filesystem permissions).

Example: The Boys S05E02 failed with "Permission denied" when writing the .sqzarr-tmp.mkv file, but was tagged as "Excluded" in the history.

## Reproduction Steps (bugs only)
1. Have a file in a directory where sqzarr lacks write permissions
2. Let the scanner pick it up and attempt transcoding
3. ffmpeg fails with "Permission denied"
4. Observe the file shows as "Excluded" in the history tab instead of "Error"


## Acceptance Criteria
- [ ] Permission denied and other I/O errors are categorized as "Error", not "Excluded"
- [ ] "Excluded" status is reserved for intentional skips (uncompressible, already meets criteria, etc.)
- [ ] Error entries are visually distinct from excluded entries in the history tab
- [ ] Error files should be retried on subsequent scans (unlike truly excluded files)
- [ ] Consider surfacing actionable errors more prominently (e.g., a notification or error count on dashboard)

## Investigation
<!-- Agent fills this during /ticket-investigate -->

**Root cause:** `handleFailure` in `internal/queue/worker.go:352-388` treats all failures identically. When `failCount >= threshold`, it calls `db.ExcludeJob()` regardless of failure type. With the default `fail_threshold = 1` (set in `internal/config/config.go:90`), *every* first failure immediately becomes "excluded" — including I/O errors like "Permission denied".

**The exact code path for a permission denied error:**
1. `transcoder.Run()` fails because ffmpeg cannot write the temp output file (permission denied).
2. `processJob()` (worker.go:258) calls `handleFailure(job, err.Error())`.
3. `handleFailure()` (worker.go:352) increments `fail_count` to 1, then checks `failCount >= threshold` (1 >= 1 = true).
4. This triggers `db.ExcludeJob()` (worker.go:367), setting status to `"excluded"` with the permission denied error as the reason.

**Why this is wrong:** The "excluded" status was designed for intentional, permanent skips — specifically the `Uncompressible` case (worker.go:276-279) where the verifier determines the file won't benefit from re-encoding. The `ExcludeJob` path in `handleFailure` was meant for files that fail *repeatedly* (the threshold exists to stop retrying hopeless files), but with threshold=1 it fires on the first failure of any kind.

**Additional consequences:**
- The scanner (`internal/scanner/scanner.go:127-133`) skips files with any status other than `""`, `failed`, or `restored`. Since "excluded" is not in the re-queue list, permission denied files are **permanently skipped** and will never be retried on subsequent scans, even after the user fixes permissions.
- The frontend (`frontend/src/components/StatusBadge.tsx:12`) shows "Excluded" with an orange badge (informational tone) rather than the red "Failed" badge, so the user has no visual cue that something needs attention.
- The History page (`frontend/src/pages/History.tsx:108`) only shows a retry button for `failed` or `cancelled` jobs, not `excluded` ones — so the user cannot manually retry a permission denied error from the UI.

**Distinction that should exist but doesn't:** There is no separation between "excluded because uncompressible" (intentional, permanent, no action needed) and "excluded because of repeated I/O errors" (needs user attention, should be retryable). Both end up calling `db.ExcludeJob()` with status `"excluded"`.

Regression Risk: low


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

Introduce an `"error"` job status for failures that need user attention (I/O errors, permission denied, etc.), distinct from `"excluded"` which remains reserved for intentional skips (uncompressible files). The `handleFailure` function should classify errors and route them to the appropriate status.

**Key changes:**

1. **New `JobError` status** in `internal/db/jobs.go` — a terminal status meaning "failed in a way that needs human intervention." Unlike `JobFailed` (transient, auto-retried), `JobError` is sticky until the user explicitly retries.

2. **Classify errors in `handleFailure`** — detect I/O errors (permission denied, disk full, read-only filesystem) and set `JobError` instead of `JobExcluded` when the threshold is hit. Only truly hopeless transcode failures (e.g., codec errors after N retries) should become `excluded`.

3. **Keep `ExcludeJob` for intentional skips only** — the direct `ExcludeJob` call for uncompressible files (worker.go:277) stays unchanged.

4. **Frontend: add `"error"` status rendering** — red badge in `StatusBadge.tsx`, show retry button for `error` jobs in History, and include `error` in the scanner's re-queue list so fixed files get picked up.

5. **Scanner: allow re-scan of `"error"` files** — add `db.JobError` to the re-queue cases in `scanner.go:129` so files are retried after the user fixes the underlying issue.

Alternative (simpler, considered and rejected): Just increase the default `fail_threshold` to 3. This would delay the misclassification but not fix it — after 3 permission denied failures the file would still become "excluded." The fundamental issue is that `handleFailure` doesn't distinguish error types.
