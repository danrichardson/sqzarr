## Description
The history tab accumulates completed jobs indefinitely. Users should be able to clear old "Done" entries from the history view to keep it manageable, but clearing history must NOT cause those files to be picked up again by the scanner and re-processed.

This is directly related to TKT-004 (files reappearing after being handled). The underlying requirement is that "processed" state must be tracked separately from "visible in history" ŌĆö history is a UI concern, processing state is a data concern.

## Investigation
<!-- Agent fills this during /ticket-investigate -->

### Revision note (2026-04-09)
The original investigation was written before TKT-004 shipped. TKT-004 added a `processed_files` table that durably tracks files reaching terminal states (done, excluded, skipped). The scanner now checks `processed_files` FIRST in `maybeEnqueue()` via `IsFileProcessed()` before checking `SourcePathStatus()`. This fundamentally changes the analysis below.

### How history clearing currently works

**API endpoint:** `POST /jobs/clear` handled by `handleClearHistory` in `internal/api/handlers.go:279`. It calls `db.ClearHistory()`.

**DB operation** (`internal/db/jobs.go:233`): Hard-DELETEs all job rows with terminal statuses `('failed','cancelled','skipped','excluded','restored')`. Also deletes associated `originals` rows first (FK cleanup). Notably, `done` and `staged` jobs are NOT deleted -- only error/skip/excluded states.

### Re-queue risk is now mitigated by processed_files

With TKT-004 shipped, the scanner's `maybeEnqueue()` calls `db.IsFileProcessed(path, size, mtime)` before anything else. If the file is in `processed_files`, it is skipped regardless of whether a `jobs` row exists. This means deleting job rows from `ClearHistory()` no longer causes re-queuing for files that were properly recorded in `processed_files`.

The worker upserts into `processed_files` when jobs reach staged/excluded state, so those files are safe from re-queuing even after history clearing.

### Remaining gaps

1. `done` and `staged` are not in the clearable statuses ŌĆö users cannot clear completed jobs from history.
2. `failed` and `cancelled` jobs are deleted by ClearHistory but may NOT have `processed_files` records (only staged/excluded/done get recorded). This is actually correct: failed/cancelled files SHOULD be re-queued.
3. The confirmation dialog says "Clear all failed, cancelled, and skipped jobs" but `restored` and `excluded` are also cleared ŌĆö the text is inaccurate.

Regression Risk: low (no schema changes needed; just expanding clearable statuses and updating UI text)


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

**Keep the DELETE-based approach for ClearHistory, expand clearable statuses, and rely on the `processed_files` table (from TKT-004) to prevent re-queuing.**

With TKT-004 shipped, the `processed_files` table already provides durable processing memory independent of job rows. The original plan to add a `hidden` column is unnecessary ŌĆö the separation of concerns already exists.

Approach:
1. Expand `ClearHistory()` to include `done` and `staged` in clearable statuses, so users can clear completed jobs.
2. Clean up originals records for cleared jobs (existing behavior, but now covers done/staged too).
3. Update `handleClearHistory` response key from `"deleted"` to `"cleared"`.
4. Update frontend confirmation text and API type to match the semantic change.
5. Add integration test proving that after ClearHistory, `IsFileProcessed` still returns true (no re-queuing).

**Why not a `hidden` column?** With `processed_files` already tracking processing state, adding a `hidden` column would be redundant complexity. The DELETE approach is simpler, keeps the jobs table lean, and the `processed_files` table already prevents re-queuing.
