## Description
The Settings panel shows the active hardware encoder (e.g., "Intel VAAPI (hevc_vaapi)") but doesn't let the user choose between options. Two things are needed:

1. **Multiple hardware encoders**: If the system has more than one hardware encoder available (e.g., VAAPI + NVENC, or multiple GPU devices), the user should be able to select which one to use from a dropdown in the settings panel.
2. **Software encoding fallback**: The user should have the option to enable software encoding (e.g., libx265) even though it's slow. This is useful as a fallback or for environments without hardware acceleration.

Investigate whether the backend already detects multiple encoders ŌĆö it may just be a UI issue of not exposing the choice. The current user only has one HW encoder so this may already be partially handled on the backend.

## Investigation
<!-- Agent fills this during /ticket-investigate -->

**Encoder detection (`internal/transcoder/encoder.go`):**
- `Detect()` probes VAAPI, VideoToolbox, and NVENC in priority order and returns the **first** match. If none match, it returns the software encoder (`libx265`).
- Four `EncoderType` constants already exist: `vaapi`, `videotoolbox`, `nvenc`, `software`.
- A `softwareEncoder()` function already exists with a complete `BuildArgs` implementation using `libx265 -crf 23 -preset medium`. Software encoding is fully implemented in the transcoder layer but is never surfaced to the user.
- **Only one encoder is returned** ŌĆö there is no `DetectAll()` or equivalent. The function short-circuits on first success.

**Config (`internal/config/`):**
- `TranscoderConfig` only has `TempDir`. There is no `encoder` or `preferred_encoder` field.
- `config.UpdateFile()` and `keySection` have no entries for encoder selection, so persisting a choice would require adding a new key under `[transcoder]`.

**API (`internal/api/`):**
- `Server` holds a single `*transcoder.Encoder` field set once at startup (`main.go:97-102`).
- `GET /status` returns `s.encoder.DisplayName` as a read-only string. No endpoint lists available encoders or accepts an encoder selection.
- `GET /config` and `PUT /config` do not include any encoder-related fields.

**Frontend (`frontend/src/pages/Settings.tsx`):**
- The Hardware section (lines 231-241) displays `status.encoder` as a read-only text label. There is no dropdown, no list of alternatives, and no mechanism to change the encoder.

**Summary:** The backend already has all four encoder implementations (VAAPI, VideoToolbox, NVENC, software) but only auto-detects one. Multi-encoder detection, user selection, persistence, and a selection UI are all missing. The software encoder is already coded but unreachable through normal operation on systems with hardware acceleration.

Regression Risk: medium


## Proposed Solution
<!-- Agent fills this during /ticket-investigate -->

1. **Add `DetectAll()` to `encoder.go`** ŌĆö returns a slice of all working encoders (probe each independently, always append software). Keep `Detect()` as a convenience that picks the first from `DetectAll()` or honors a config override.

2. **Add `encoder` key to `TranscoderConfig`** ŌĆö a string field (`"vaapi"`, `"nvenc"`, `"videotoolbox"`, `"software"`, or `""` for auto). Persist via `config.UpdateFile` under `[transcoder]`. Empty/missing means auto-detect (current behavior, backward-compatible).

3. **New API endpoints:**
   - `GET /encoders` ŌĆö returns the list from `DetectAll()` with `type`, `display_name`, and `active` boolean.
   - Extend `PUT /config` to accept `encoder` field. On change: validate the requested encoder is in the detected list, update `TranscoderConfig.Encoder`, persist to TOML, and swap the active encoder on the `Transcoder` and `Server` structs.

4. **Frontend:** Replace the read-only Hardware label with a dropdown populated from `GET /encoders`. Selection triggers `PUT /config` with the new encoder type. Show a badge for the currently active encoder.

5. **Hot-swap safety:** If a job is currently running, the encoder change takes effect on the *next* job (the `Worker` already gets its encoder from the `Transcoder`, which holds a pointer). No restart required.
